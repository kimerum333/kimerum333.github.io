## 서언
새로 토이프로젝트 만지는 김에 접한 uv.<br>
한번 써보는 김에 제대로 알고 써보자고 생각해서, 좀 주변 맥락도 같이 조사해봤다.

# UV란 무엇인가
파이썬 생태계의 최신 환경관리 툴.

## 다른 대체재들(시간순)
### requirements.txt
가장 클래식한 해결책.<br>
의존성 목록을 기본적인 txt 파일의 형태로 저장한 것.<br>
venv등의 도구로 이 파일을 읽어들여, pip를 이용해 폴더 단위 가상환경을 설치하는 식으로 동작한다.<br>

단점으로는, 내가 직접 설치한 라이브러리의 버전만 추적하기 때문에, 해당 라이브러리의 파생 의존성들의 버전은 모르게 된다.<br>
파생 의존성들끼리 싸우기 시작하면 답도 없다.

### anaconda
폴더단위 가상환경이 아니라, 로컬환경을 통째로 갈아버린다. 덕분에, 다른 머신에 프로젝트를 옮길 때 불편하다.<br>
데이터 사이언스에 쓰는 무거운 라이브러리들을 쓰기 위한 해법으로, 개인적으로는 단순 python 어플리케이션을 위해서는 적합하지 않다고 본다.

### poetry
requirements의 단점을 해결한, pyproject.toml이라는 표준 규격 의존성 목록 파일을 기반으로 하는 현대적인 관리 도구.<br>
라이브러리 간의 버전 엉킴을 방지하는 '의존성 해결' 능력을 갖고 있다. 패키징과 배포 기능까지 포함하고 있다. <br>

UV가 등장하기 전까지 사실상 파이썬 프로젝트의 표준(De facto)<br>
단, 프로젝트가 커질수록 의존성을 계산하는 속도가 현저히 느려진다는 단점이 있다.<br>

계산 완료된 파생 의존성 스냅샷은 poetry.lock 의 형태로 보관.

## 그럼 uv는 어떻길래
### 의존성 해결 능력
Poetry의 최대 장점이었던 pyproject.toml 과 poetry.lock을 그대로 가져와서, pyproject.toml과 uv.lock으로 사용. 의존성 문제를 그대로 해결했는데... <br>

Rust로 돌리는 데다, 다운로드와 의존성 계산을 병렬로 동시에 시켜서 속도를 빠르게 해버렸다.

### 하나의 도구로 통합
uv이전에는 각 상황에 맞는 도구를 따로 써야했다. uv는 uv가 다 한다.
- 파이썬 버전 설치 및 관리 : pyenv => uv python install
- 가상환경 생성 : venv => uv venv
- 패키지 설치 : pip => uv add , uv sync

그 외 편의성 기능들도 있다.
- uvx : 노드 진영의 npx 같은 기능. '설치 없는 실행'(임시 공간에 패키지 설치해서 다 쓴 뒤 삭제)
- uv run : makefile과 비슷. 미리 지정된 명령어를 수행한다. 수행하기 전에 의존성을 전부 완비해준다.

### 글로벌 캐시 기반의 설치
uv는 여러 프로젝트에서 동일한 라이브러리를 쓸 때, global cache 에 이를 설치한다.<br>
이후각 프로젝트에서는 링크만 붙여서 하나를 재활용한다.<br>
일종의 싱글톤 개념? 용량이 절감되고, 설치 속도가 빨라진다.

## 실전 용례
내 개발 환경인 wsl 위에서 uv를 써서 첫 FastAPI프로젝트를 세팅하는 시나리오를 작성했다.<br>
명령어는 시간이 흐르면 바뀌는 경우가 많으므로 흐름을 잡는 용도로 사용하도록 하자.<br>
언제나 가장 강력한 레퍼런스는 공식 문서이니.

### 1. uv의 설치
```sh
curl -LsSf https://astral.sh/uv/install.sh | sh
```
### 2. 특정 파이썬 버전 설치.

```sh
# uv가 관리하는, 격리된 파이썬 버전을 설치해준다.
uv python install 3.12
```
### 3. 새 프로젝트 생성 및 init
```sh
# pwd에 my-fastapi-app이란 이름으로 디렉터리 만들고 main.py나 gitignore, .git 등 기본적인 뼈대를 잡아준다. 의존성 관리를 위한 pyproject.toml도 생긴다.
uv init my-fastapi-app
```

### 4. 프로젝트에 새 패키지 추가
```sh
cd my-fastapi-app
# 의존성 추가. 만약 해당 디렉터리에 .venv가 없다면 먼저 만든다.
uv add fastapi
# .venv 안에 파일이 설치되며, .toml도 업데이트된다.
```
#### 4-1. Dev 의존성을 따로 관리하고 싶다면
```sh
# --dev 옵션을 통해 
uv add --dev pytest
# .toml 의 dev섹션에 따로 업데이트된다.
```
#### 4-2. 버전을 명시해 설치하고 싶다면
```sh
# 정확한 명시
uv add "fastapi==0.115.0"
# 최소 버전 지정
uv add "fastapi>=0.115.0"
# 특정 범위 지정 (0.110.0 이상, 0.116.0 미만)
uv add "fastapi>=0.110.0,<0.116.0"
# 호환되는 최신 버전(0.115.x 중 최신 설치)
uv add "fastapi~=0.115.0"

# 만약 버전을 변경하고 싶다면, 그냥 새 버전을 add하라. 기존 버전을 자동으로 지워준다.
```

#### 4-3. remove
```sh
uv remove fastapi
uv remove --dev fastapi
``` 

### 5. 다른 머신에서 설치시
```sh
# 기본 명령어. .toml 을 바탕으로 전부 설치한다.
uv sync
# prod 환경용. dev의존성은 제외하고 설치한다.
uv sync --no-dev
```

### 6. Run
#### 6-1. 나만의 Run 명령어 정의
```toml
# .toml 파일 하단에 이렇게 직접 작성
# 명령어 = "폴더명.파일명:함수명" 형태로 작성
[project.scripts]
start-server = "app.main:run_app"
test-all = "pytest"
hello = "scripts.greetings:say_hello"
```

## 결언
poetry의 장점을 계승하고, npm의 편리함을 더한 듯한 편리한 도구다.<br>
하지만, 모든 면에서 우월한 도구라고 하기는 힘들다.<br>

poetry의 wheel을 만들어 배포하는 빌드 시스템의 안정성은 아직 uv가 따라오지 못한 영역이다. 전 세계에 배포할 고성능 라이브러리를 만드는 슈퍼 개발자라면 poetry 를 썼겠지만, 나같은 일개 서버 개발자에게는 uv 쪽이 쓰기 더 편리한 것 같다.
